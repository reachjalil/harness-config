# أدوات Harness config

`harnessc` هو التنفيذ المعياري لـ Harness config. وجوده يسمح للمستودعات بالتحقق من شكل الملفات، ومعاينة التفعيل، وتحقيق إسقاطات النسخ دون كتابة أداة مخصصة أولاً.

أي تنفيذ آخر يحقق مطابقة الأدوات صالح بالقدر نفسه. المعيار يعرّفه شكل المستودع وعقد التفعيل، لا ملف تنفيذي واحد.

## الخصوصية وقياس الاستخدام

لا يجمع Harness config قياس استخدام.

لا ترسل واجهة `harnessc` تحليلات أو أحداث استخدام أو مسارات ملفات أو أسماء مستودعات أو سجل أوامر أو معرفات أجهزة أو تقارير أخطاء.

يعمل التفعيل والتحقق والتخطيط محلياً على الملفات داخل مستودعك. لا تنفذ واجهة سطر الأوامر طلبات شبكة أثناء التشغيل العادي.

## الأوامر

```bash
harnessc
harnessc init
harnessc validate
harnessc explain <path>
harnessc activate
harnessc extension activate
harnessc plan
```

- `harnessc` بدون أمر يتحقق من أقرب تكوين مستودع ويطبع مسار البيان المكتشف مع الخطوات التالية المقترحة.
- `harnessc init` ينشئ البيان المختار، افتراضياً `./.harness/harness.toml`، ومجلدات الموارد التقليدية أو المخصصة تحت جذر مصدر الموارد المهيأ، و`.harnessIgnore` و`.harnessMutable` عند التطبيق مع `--yes`. يصرّح البيان الابتدائي المولّد بـ `[[resources]] path = "./.harness/resources"` صراحة.
- `harnessc validate` يفحص دعم الإصدار، والمسارات المحلية داخل المستودع، وخرائط الأهداف، وصيغة تجاهل الإسقاط، وصيغة تصريحات الملفات القابلة للتعديل، وأوراق الموارد القابلة للتركيب، ومعاملة الرابط الرمزي كمدخل ورقي، ومشكلات تركيب أو نسخ `dir`.
- `harnessc explain <path>` يشرح كيف يشارك مسار مصدر أو مخرج في خطة الإسقاط الحالية، بما في ذلك مسارات المصدر الفائزة، وجذور المصدر المهيأة، ومخرجات `dir`، والتشخيصات الحاجزة، وقرارات `.harnessIgnore`، وقرارات ملكية `.harnessMutable`. يتضمن مخرج JSON آثار تجاهل المصدر ومخرجات الهدف حتى يستطيع المستدعي تمييز استبعاد جذر المستودع، وإعادة تضمين أعمق محلية للمصدر، وإعادة تضمين منطقية محلية لملف التعريف، والحد النهائي في مخرجات الهدف.
- `harnessc activate` يجري تجربة لإسقاط التفعيل ويعرض الإنشاءات والتحديثات وطلبات الإزالة والملفات المحتفظ بها والملفات المتخطاة لأنها قابلة للتعديل والمدخلات غير المُدارة المحفوظة قبل الكتابة. افتراضياً يبلغ عن روابط الهدف الرمزية التي تشغل مسارات مسقطة كتعارضات؛ مرر `--replace-target-symlinks` أو اضبط `[activation].targetSymlinks = "replace"` لاستبدال الرابط نفسه.
- `harnessc extension activate` يشغل الامتدادات المسجلة. استخدم `--extension <id>` لتشغيل امتداد مصرح به واحد، أو `--all` لتشغيل كل امتداد مصرح به ومدعوم.
- `harnessc plan` خطة تهيئة أو تطبيق للقراءة فقط. ليست معاينة إسقاط. استخدم `harnessc activate` بدون `--yes` لمعاينة الإسقاط.

`init` و`activate` و`extension activate` تشغيلات تجريبية ما لم يمرر `--yes`.

أمثلة فحص شائعة:

```bash
harnessc explain .agents/skills/review/SKILL.md
harnessc explain AGENTS.md
harnessc explain .harness/local/resources/skills/review/SKILL.md
```

تحفظ المدخلات غير المُدارة في الأهداف افتراضياً. استخدم `--remove-unmanaged` عندما يجب تنظيف الهدف ليطابق المصادر المهيأة؛ واستخدم `--keep-unmanaged` لجعل الافتراض صريحاً.

يمكن تجاهل واجهات التشغيل المولّدة مثل `.agents` و`.claude` و`.cursor` و`.gemini` في Git عندما تكون قابلة للإعادة من `.harness`. المشاريع التي تفعل ذلك يجب أن تحتفظ بمسار تمهيد متتبع مثل ملاحظة تعليمات في الجذر، أو خطوة إعداد في README، أو سكربت package، يخبر المستخدمين والوكلاء بتشغيل التحقق والتفعيل بعد جلب نسخة جديدة.

مدخلات `.gitignore` الموصى بها بعد ترحيل كامل هي:

```gitignore
# Harness-generated live surfaces
.agents/
.claude/
.cursor/
.gemini/

# Private Harness overlays
.harness/local/
```

أبق مصدر Harness المشترك متتبعاً: `.harness/harness.toml`، و`.harness/resources/**` المشتركة، و`.harness/dir/**` عند استخدامها، وتصريحات `.harnessIgnore` و`.harnessMutable`. لا تضف `.harness/` كتجاهل واسع إلا إذا كان المستودع يختار قصداً عدم مشاركة فهرس المصدر.

ينطبق التنظيف فقط على الأهداف التي ما زالت مصرحاً بها في البيان المختار. بعد إزالة تصريح هدف، لا يعود `harnessc activate` الأساسي يفحص ذلك المجلد أو ينظفه. نظفه أولاً باستخدام `--remove-unmanaged`، أو استخدم مسار عمل أعلى لحالة التفعيل يستطيع التوفيق مع الأهداف اليتيمة.

المسار الافتراضي للبيان هو `./.harness/harness.toml`. عندما لا يمرر `--root` ولا `--config`، يبحث `harnessc` صعوداً من الدليل الحالي عن ذلك البيان. مرر `--config <path>` للتحقق أو التخطيط أو التهيئة أو التفعيل أو تشغيل الامتدادات على ملف TOML آخر محلي داخل المستودع. يكتب `harnessc init --resources-path <path>` مدخل `[[resources]]` واحداً في البيان وينشئ مجلدات الموارد أسفل جذر المصدر المهيأ. تختار استدعاءات الأداة مسارات البيان؛ أما المسارات داخل البيان فتبقى محلية داخل المستودع، لا نسبية إلى دليل ملف البيان.

خطة التفعيل هي أيضاً عرض الملكية للمشغل. الملفات المُدارة مخرجات إسقاط يملكها المستودع، والمدخلات غير المُدارة حالة هدف قائمة خارج الإسقاط، والمدخلات القابلة للتعديل ملفات هدف زُرعت من المصدر لكنها الآن مملوكة لوقت التشغيل.

تقارن الملفات المُدارة مباشرة مع إسقاط المصدر الحالي: إذا اختلف الهدف، يبلغ `harnessc activate` عن `update`، ويكتب تطبيق التفعيل الهدف ببايتات المصدر الحالية. الملفات القابلة للتعديل المعلنة في `.harnessMutable` تُنشأ مرة واحدة من المصدر وتتخطى في التفعيلات اللاحقة لأن بايتات الهدف المباشر مملوكة لوقت التشغيل. استخدم `--force-mutable` لإعادة إسقاطها من المصدر.

لا تُتبع روابط الهدف الرمزية. إذا شغل رابط رمزي في الهدف مساراً يحتاج الإسقاط إلى الكتابة فيه، يبلغ `harnessc activate` عن `harness.target_symlink_conflict` ويرفض `--yes` افتراضياً. اختر إحدى سياسات الاستبدال الصريحة عندما يكون استبدال الرابط نفسه مقصوداً:

```toml
[activation]
targetSymlinks = "replace"
```

أو لتشغيل واحد:

```bash
harnessc activate --yes --replace-target-symlinks
```

يتبع سلوك نظام الملفات تجميد إصدار v1:

- تعامل الروابط الرمزية كمدخلات ورقية ولا تُتبع.
- روابط الهدف الرمزية التي تشغل مسارات مسقطة تعد تعارضات ما لم يُختر الاستبدال بسياسة البيان أو `--replace-target-symlinks`.
- تحفظ المدخلات غير المُدارة ما لم يختر `--remove-unmanaged`.
- تحفظ ملفات `.harnessIgnore` و`.harnessProfile` في مخرجات الهدف كضوابط محلية.
- يتقارب التفعيل المتكرر بالمدخلات نفسها إلى `keep` للملفات المُدارة و`mutable` للملفات التي يملكها وقت التشغيل.
- الأهداف المتداخلة أو الأهداف التي تتصادم مع جذور مصدر مهيأة تشخيصات.

مسارات عمل الاختيار، وسلوك الأسواق، ومراجعة تعديلات الهدف، والالتقاط، وآراء المنتج الأخرى مكانها فوق `harnessc`.

## تركيب ونسخ `dir`

يؤدي التصريح بمدخل واحد أو أكثر من `[[dir]]` في البيان المختار إلى تفعيل جذور مصدر `dir` المرتبة. يخطط `harnessc activate` ويطبق مخرجات `dir` جنباً إلى جنب مع إسقاط الهدف:

```toml
[[dir]]
path = "./.harness/dir"

[[dir]]
path = "./.harness/local/dir"
```

```text
.harness/dir/
  AGENTS.md/
    .harnessComposable
    100_intro.md
    200_rules.md
  CLAUDE.md/
    .harnessComposable
    .harnessRef                       # ../AGENTS.md
    150_claude.md
  .github/
    copilot-instructions.md/
      .harnessComposable
      100_intro.md
  .claude/
    settings.json              # copy mode (no marker)
  notes/
    01_dev_intro.md            # copy mode (no marker)
```

داخل كل مصدر `[[dir]]`، المجلدات التي تحتوي ملف `.harnessComposable` فارغاً هي أوراق قابلة للتركيب: تدمج أجزاؤها ذات البادئة الرقمية، مثل `100_intro.md` و`200_rules.md`، بالترتيب لإنتاج ملف مخرج واحد نسبي للمستودع. والمجلدات التي لا تحمل العلامة هي مجلدات نسخ: تنسخ ملفاتها وملفاتها المتداخلة إلى المسار النسبي المطابق في المستودع. وتنسخ الملفات المفردة في أي عمق أيضاً.

يمكن استخدام علامة `.harnessComposable` نفسها تحت مصدر موارد مهيأ. في ذلك الموقع يركب ملف مورد مسقطاً واحداً داخل كل هدف مصرح به؛ وليس مخرج `[[dir]]` نسبياً للمستودع.

تستورد ملفات `.harnessRef` داخل ورقة قابلة للتركيب أجزاء ورقة أخرى. ترتب الأجزاء المستوردة والمحلية معاً، وتُبقي الأرقام المكررة كل الأجزاء المطابقة، وتبلغ الدورات أو أهداف `.harnessRef` المفقودة كأخطاء.

تطبق قواعد `.harnessIgnore` في جانب المصدر أثناء جمع `dir`، بما في ذلك القواعد داخل ورقة `.harnessComposable` والقواعد داخل مصدر `[[dir]]` مخصص خارج `./.harness`. تجاهل الحاوية يتخطى كل مخرجات `dir` تحتها، وتجاهل الورقة يتخطى ذلك المخرج، وتجاهل جزء يستبعد ذلك الجزء من التركيب. ويمكن لملفات `.harnessIgnore` في مخرجات الهدف أن ترشح مخرجات `dir` بحسب مسار المخرج النهائي بعد معرفة بنية المخرج المرشحة؛ وتقيّم قواعد مخرجات الهدف بعد قواعد المصدر وملف التعريف المحلية، لذلك تشكل الحد النهائي لتلك الشجرة الفرعية. تُتجاهل ترويسات الهدف المقيّدة في هذا الوضع. ولا تنسخ علامة `.harnessComposable` نفسها إلى أي مخرج.

تستخدم طبقات ملفات التعريف محددات `.harnessProfile` وطبقات مصدر `.harnessProfileRoot`. يطبق `.harnessProfile` في الجذر عالمياً؛ وتطبق محددات الهدف أو المخرج مثل `.agents/skills/.harnessProfile` على تلك الشجرة الفرعية فقط. يجب أن يعيش `.harnessProfileRoot` تحت `.harness` أو مصدر موارد مهيأ أو مصدر `dir` مهيأ؛ وعندما يكون نشطاً، تركب محتوياته إما جذر المصدر الأب، للعلامات المباشرة داخل مصدر الموارد أو جذر `dir`، أو الدليل الأب، لجذور ملفات التعريف المحمولة المتداخلة داخل عنصر مورد أو شجرة `dir`، أو `.harness`، لمجلدات الحزم.
لا يجوز تداخل جذور ملفات التعريف داخل جذور ملفات تعريف أخرى. تطابق ملفات `.harnessIgnore` المحلية لملف التعريف تلك المسارات المنطقية، بما فيها أوراق `.harnessComposable`. يكتشف تخطيط `dir` محددات ملفات التعريف في الهدف أو المخرج من المخرجات المرشحة الأساسية والموجودة في ملف التعريف فقط قبل حساب مجموعة مخرجات `dir` النهائية.

## تخصيص المطور الواحد

للمطورين المنفردين أو الفرق التي تريد تجربة محلية دون تغيير المصادر المشتركة المراجعة، يعمل `harnessc` مع جذور مصدر إضافية مرتبة. نمط عملي:

```toml
[[resources]]
path = "./.harness/resources"

[[resources]]
path = "./.harness/local/resources"

[[dir]]
path = "./.harness/dir"

[[dir]]
path = "./.harness/local/dir"
```

لا تطلب واجهة CLI وجود هذه المسارات. يمكن للمشاريع أن تتجاهل `.harness/local/` في التحكم بالإصدارات، أو تتتبعه، أو تولده، أو تستخدم مسارات مختلفة. تستبدل الجذور اللاحقة مخرجات الموارد أو `dir` ذات المسار الدقيق من الجذور السابقة؛ استخدم `harnessc explain <path>` لفحص سبب وجود مسار مصدر أو مخرج معين أو تجاهله أو تجاوزه أو تركيبه.

عندما تكون `.harness/local/` متجاهلة في Git، تستطيع ملفات البيان المشتركة مع ذلك التصريح بها كجذر لاحق اختياري. الجذور المحلية الغائبة لا تضيف أي ملفات محلية؛ والجذور المحلية الموجودة تستطيع تجاوز مخرجات الموارد أو `dir` ذات المسار الدقيق لذلك المطور.

مسارات مخرجات `dir` التي تقع تحت مسار `[[targets]]` مصرح به تدمج في إسقاط ذلك الهدف، وتشغيل التفعيل مرة ثانية يتقارب إلى إجراءات `keep` لهذه الملفات، بما في ذلك تنظيف المدخلات غير المُدارة في الهدف. أما مخرج `dir` الذي سيستبدل جذر هدف أو يحتويه، مثل مخرج `dir` عند `.claude` عندما يكون `./.claude` هدفاً مصرحاً به، فيبلغ عنه كـ `harness.dir_output_target_overlap`.

## الامتدادات

يأتي `harnessc` مع سجل امتدادات للتوافق المستقبلي. لا يحمل هذا الإصدار أي تنفيذ امتداد مدمج؛ الأدوات التي تصرّح بـ `[extensions.<id>]` لمعرفات غير مدعومة ترى تشخيصاً معلوماتياً بدلاً من سلوك. سطح تركيب ونسخ `dir` أعلاه جزء من تفعيل النواة، لا امتداد.

```toml
[extensions.example]
version = 1
activation = "explicit"
```

تملك النواة `version` و`activation`. أما الحقول الخاصة بالامتداد فتعود إلى تنفيذ الامتداد المسجل. يشغل `harnessc extension activate` المجرد الامتدادات المهيأة بـ `activation = "auto"` فقط.

## مساعدات TypeScript

تستطيع المحررات وسكربتات CI والأدوات الداخلية تضمين السلوك نفسه عبر `@harnessconfig/core`:

```ts
import {
  applyHarnessActivation,
  loadHarnessIgnoreMatcher,
  parseHarnessConfigToml,
  planHarnessActivation,
  resolveHarnessPaths,
  validateHarnessConfig,
} from "@harnessconfig/core";

const paths = resolveHarnessPaths(process.cwd());
const config = parseHarnessConfigToml(rawToml);
const ignore = await loadHarnessIgnoreMatcher(paths.root);
const validation = await validateHarnessConfig(paths.root);
const activationPlan = await planHarnessActivation(paths.root);
const dryRun = await applyHarnessActivation(paths.root);
```

## فحوصات المدقق

يجب على المدقق المطابق أن:

- يستخدم مسار `--root` الممرر، أو دليل العمل الحالي عندما لا يمرر root، كجذر للمستودع. يجب على الاستدعاءات المتداخلة تمرير `--root`.
- يحلل البيان المختار، افتراضياً `./.harness/harness.toml`، ويرفض المدخلات المشوهة بتشخيصات واضحة.
- يرفض إصدارات المعيار المستقبلية غير المدعومة.
- يتحقق من مسارات مصادر الموارد المهيأة ويرفض تصريحات الموارد بحسب النوع داخل البيان.
- يتحقق من أن كل مدخل `[[targets]]` يحتوي مساراً محلياً داخل المستودع فقط، ويشير إلى ما تحت جذر المستودع، ولا يتداخل مع جذور مصدر مهيأة.
- يحلل `.harnessIgnore` بقواعد الجذر والمحلية للمصدر والمحلية لملف التعريف والمحلية لمخرجات الهدف وفق مراحل الأولوية المعيارية. ويحلل `.harnessMutable` منفصلاً للملفات التي يملكها وقت التشغيل وتُنشأ مرة واحدة.
- يحل محددات `.harnessProfile` وطبقات `.harnessProfileRoot` قبل الإسقاط، بما في ذلك المسار التمهيدي/النهائي في `dir` لمحددات المخرجات.
- يعرض إجراءات `create` و`update` و`remove` و`keep` و`preserve` و`mutable` قبل أي كتابة.
- يتحقق من أن التفعيل المتكرر على مدخلات غير متغيرة يتقارب إلى شجرة الهدف نفسها للملفات المُدارة، ويترك الملفات القابلة للتعديل كما هي.
- يبلغ عن الأهداف المصرح بها منفصلة عن مجلدات المصدر طويلة العمر.

## المخرجات

المخرج الافتراضي تقرير طرفية موجز بالمسارات والخطورة والإصلاحات المقترحة. يجوز لمخرج الطرفية البشري استخدام ANSI color للعناوين وخطورة التشخيص وأنواع الإجراءات عندما يدعم مجرى المخرجات اللون. يجب أن تتجنب التطبيقات اللون في المخرجات المعاد توجيهها، وأن تحترم `NO_COLOR`، وأن تبقي مخرجات `--json` خالية من ANSI escapes للأتمتة وCI.
