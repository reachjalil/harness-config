# تطبيق Harness config

يوضح هذا الدليل مسارين: الإعداد من الصفر، عندما لا توجد مجلدات وكلاء مباشرة، والترحيل عندما يحتوي المستودع بالفعل على `.claude/` أو `.cursor/` أو `.agents/` أو ما يشبهها.

## الإعداد من الصفر

يبدأ Harness config v1 من عقد مصدر صغير:

1. أنشئ `./.harness/harness.toml` مع `version = 1`، أو اختر مسار بيان آخر محلياً داخل المستودع ومرره صراحة إلى الأدوات.
2. أضف مجلدات وملفات الموارد تحت جذور مصدر `[[resources]]` صريحة، غالباً `.harness/resources`، مثل `.harness/resources/skills` أو `.harness/resources/rules` أو `.harness/resources/hooks.json`.
3. صرّح بكل هدف إسقاط صراحة في البيان المختار.
4. استخدم `.harnessIgnore` لإبقاء ملفات المصدر فقط خارج الأهداف المباشرة، واستخدم `.harnessMutable` للملفات التي يملكها وقت التشغيل ويجب زرعها مرة واحدة. عادة يكون الملف القابل للتعديل قالب مصدر لحالة هدف محلية، لا تكويناً مشتركاً طويل العمر.
5. شغّل التفعيل كتجربة قبل كتابة مجلدات الهدف.

لإعداد أول صغير، اعرض البيان المختار وشجرة المصدر معاً:

```toml
version = 1

[[resources]]
path = "./.harness/resources"

[[targets]]
path = "./.agents"
```

```text
AGENTS.md                         # tracked activation/root instructions
.harnessIgnore
.harnessMutable
.harness/
  harness.toml
  resources/
    README.md
    skills/
      review/
        SKILL.md
.agents/                          # generated after activation
```

احتفظ بـ `AGENTS.md` أو `CLAUDE.md` أو ملفات تعليمات جذرية مشابهة كملفات متتبعة عادية عندما تكون بسيطة ومتماسكة بالفعل. انقلها إلى `[[dir]]` فقط عندما تجعل التوليد أو التركيب أو ملفات التعريف أو الطبقات المحلية المستودع أسهل للفهم.

`harnessc` هو التنفيذ المعياري لهذا المسار:

```bash
harnessc init
harnessc init --yes --resource skills --target ./runtime/agent
harnessc validate
harnessc explain .agents/skills/review/SKILL.md
harnessc activate
harnessc activate --yes
```

## ترحيل مستودع قائم

يعتمد المستودع الذي لديه واجهات تشغيل مثل `.claude/` أو `.cursor/` Harness config تدريجياً. شكل الترحيل مهم: يجب أن تصبح جذور المصدر المصرح بها هي المدخل الرسمي، لا الهدف.

التسلسل الموصى به:

1. **خذ لقطة من الأهداف القائمة.** التزم بالمجلدات المباشرة الحالية أو انسخها إلى فرع. التطبيق قابل للعكس، لكن وجود خط أساس معروف يجعل المراجعة أسهل.
2. **انقل المحتوى طويل العمر إلى تخطيط موارد واضح.** في أول ترحيل كامل، فضّل جذراً واحداً هو `./.harness/resources` واجمع داخله بحسب الفائدة: سير العمل، أو الاستراتيجية، أو الفريق، أو الوضع، أو مجموعة الوكلاء، أو مساحة المنتج، أو الحزمة. يتحول معظم محتوى `.claude/skills/foo/` إلى `./.harness/resources/skills/foo/`. والملفات التي تختلف لوكيل واحد فقط تنتقل إلى مجلد تجاوز العنصر المطابق، مثل `./.harness/resources/skills/foo/.claude/`. أما ملفات جذر الهدف مثل `.claude/settings.json` أو `.claude/hooks.json` فمكانها المسار المطابق المشتق من الهدف تحت جذر الموارد، مثل `./.harness/resources/.claude/settings.json` أو `./.harness/resources/.claude/hooks.json`. أضف جذور موارد مهيأة أخرى فقط لفهارس اهتمامات اختيارية مثل الاختبار أو النشر أو عمل واجهة المستخدم، أو لحدود ملكية منفصلة، أو لتخصصات يختارها ملف تعريف، أو لطبقات محلية/خاصة.

   ```toml
   [[resources]]
   path = "./.harness/resources"

   [[resources]]
   path = "./.harness/local/resources"

   [[targets]]
   path = "./.agents"

   [[targets]]
   path = "./.claude"
   ```

   ```text
   .harness/
     resources/
       README.md
       .claude/
         settings.json
         .harnessMutable
       skills/
       prompts/
       rules/
       plugins/
     local/
       resources/
   ```

   يجعل هذا الاقتران مراجعة الترحيل ملموسة: يرى المراجعون أي جذر مصدر مشترك سيُسقط، وأي ملفات على مستوى الهدف ستُزرع، وأي جذر مصدر محلي خاص أو تجريبي. ومع نضج Harness config، تستطيع الفرق فصل الاهتمامات إلى جذور إضافية مثل `./.harness/resources-testing` أو `./.harness/resources-deployment` أو `./.harness/resources-ui` وجمعها مع طبقات ملفات التعريف أو تعليمات `dir` الخاصة بملف تعريف.
3. **صرّح بالأهداف في البيان المختار.** أضف مدخل `[[targets]]` لكل واجهة تشغيل تريد إعادة توليدها. لا يستلم الهدف الإسقاطات إلا عندما يظهر هنا. وصرّح بكل مصدر مشترك بمدخل `[[resources]]` صريح.
4. **اكتب `.harnessIgnore` و`.harnessMutable` عن قصد.** السجلات والملفات المؤقتة وبيانات التعريف الخاصة بالأداة و`metadata.toml` للمهارات غالباً تنتمي إلى قواعد التجاهل لأنها لا يجب أن تعبر حد الإسقاط. أما الملفات التي يكتبها وقت التشغيل مرة أخرى، مثل الصلاحيات والإعدادات المحلية والأوامر المتعلّمة، فتنتمي إلى `.harnessMutable` عندما يجب أن يزرعها فهرس المصدر مرة واحدة ثم يملكها وقت التشغيل في الهدف بعد ذلك. انسخ هذه الملفات كبذور داخل `.harness` قبل إعلانها قابلة للتعديل، حتى تستلم النسخ الجديدة إصداراً أولياً.
   تعيش القواعد العامة للمستودع عادة في `./.harnessIgnore`؛ ويمكن لقواعد موارد أو `dir` محددة أن تعيش في ملفات `.harnessIgnore` محلية للمصدر، ويمكن لتفضيلات المخرجات المحلية للمستخدم أن تعيش في ملفات مخرجات هدف مثل `runtime/agent/skills/foo/.harnessIgnore`. تفيد ملفات مخرجات الهدف عندما تكون واجهة التشغيل المباشرة داخل `.gitignore` ويحتاج المطور إلى حد محلي مؤقت؛ أما القواعد المشتركة فيجب أن تعيش في المصدر. تتبع الأولوية عمق الدليل المنطقي، لذلك تستطيع قواعد المصدر أو ملف التعريف الأعمق إعادة تضمين مسارات مختارة، بينما تبقى قواعد مخرجات الهدف الحد النهائي.
5. **أضف طبقات ملفات التعريف فقط حيث توضح الملكية.** ضع `.harnessProfileRoot` تحت `.harness` أو تحت مصدر موارد مصرح به أو تحت مصدر `dir` مصرح به للحزم الاختيارية أو الطبقات الشخصية، واخترها بملفات `.harnessProfile` في جذر المستودع أو في مخرجات الهدف. تستطيع ملفات `.harnessIgnore` المحلية لملف التعريف إخفاء ملفات أساس أو أجزاء قابلة للتركيب لذلك الملف، وتُقيّم عند موقع طبقة ملف التعريف المنطقي. استخدم ملفات التعريف أولاً كأوضاع قابلة للتبديل عبر مجموعات الموارد، وكطبقات ملفات فقط عندما تضيف أو تستبدل محتوى فعلياً.
6. **شغّل تجربة، واشرح، وراجع، ثم طبّق.** يطبع `harnessc activate` الخطة دون كتابة. استخدم `harnessc explain <path>` لمسار مصدر أو مخرج يحتاج فحصاً، ثم راجع إجراءات `create` / `update` / `remove` مقابل اللقطة وأعد التشغيل مع `--yes`.
7. **أعد تشغيل التفعيل.** يجب أن تتقارب تجربة ثانية على مدخلات غير متغيرة إلى `keep` للملفات المُدارة و`mutable` للملفات التي يملكها وقت التشغيل. إن لم يحدث ذلك، فما زالت شجرة المصدر تنحرف عن الهدف؛ صحح هذا الانحراف قبل الاعتماد على المعيار.

بعد الترحيل، تصبح المجلدات المباشرة مشتقة: يمكن حذفها وإعادة توليدها من جذور المصدر المصرح بها مع البيان في أي وقت. ويمكن للفرق أيضاً تجاهل واجهات التشغيل المباشرة في Git عندما تريد مساحة أكبر للتجارب المحلية أو حالة وقت التشغيل أو ملفات مؤقتة خاصة بالأداة. هذه المقايضة مقصودة: تحدث المراجعة في `.harness` والبيان المختار، وتبقى ملفات الهدف المُدارة قابلة للإعادة، وتحفظ ملفات `.harnessMutable` في الهدف حالة وقت التشغيل خارج شجرة المصدر الرسمية.

## توصيات `.gitignore`

استخدم `.gitignore` فقط بعد أن يصبح مصدر الحقيقة واضحاً:

- **تتبع مصدر Harness المشترك.** التزم بـ `.harness/harness.toml`، و`.harness/resources/**` المشتركة، و`.harness/dir/**` عند استخدامها، وتصريحات `.harnessIgnore` و`.harnessMutable` اللازمة لإعادة توليد المخرجات.
- **تجاهل واجهات التشغيل المولّدة بعد التقارب.** بعد أن يتقارب التفعيل وتمثل كل الموارد الدائمة داخل `.harness`، يمكن تجاهل مجلدات مثل `.agents/` و`.claude/` و`.cursor/` و`.gemini/`. أبق تعليمات تفعيل متتبعة، مثل ملاحظة تعليمات في الجذر، أو خطوة إعداد في README، أو سكربت package يشغل التحقق والتفعيل.
- **تجاهل طبقات المطور المحلية عند الحاجة.** استخدم `.harness/local/` للمهارات والمطالبات والتجارب وطبقات `dir` الخاصة، ثم أضفه إلى `.gitignore` عندما لا ينبغي مشاركة تلك الملفات.
- **لا تعتمد على ضوابط هدف متجاهلة في أول تفعيل.** ملف `.harnessIgnore` أو `.harnessProfile` داخل مخرجات الهدف لا يشارك إلا بعد وجوده في المخرج المولّد. ضع حدود أول تفعيل المشتركة في ملفات `.harnessIgnore` المحلية للمصدر أو في `.harnessIgnore` في جذر المستودع.

مثال بعد ترحيل كامل:

```gitignore
# Harness-generated live surfaces
.agents/
.claude/
.cursor/
.gemini/

# Private Harness overlays
.harness/local/
```

لا تتجاهل `.harness/` كلها؛ فهذا يخفي البيان والمصدر المراجع اللازمين لإعادة توليد واجهات التشغيل المباشرة.

## أخطاء شائعة

- **معاملة مجلد مباشر كمصدر وهدف معاً.** لا توجه مدخل `[[targets]]` إلى مجلد تعدله مباشرة أيضاً. سيبلغ التفعيل التالي عن انحراف أو يكتب فوق التعديلات المباشرة. إذا كان يجب أن يبقى المجلد مصدراً حالياً، فاتركه خارج `[[targets]]`.
- **نسيان التصريح بهدف.** لا تُسقط الموارد إلا إلى الأهداف المصرح بها. قد يحتوي مستودع على `./.harness/resources/skills/foo/` و`.claude/` على القرص ومع ذلك يرى "no creates" لأن `./.claude` غير موجود في `[[targets]]`.
- **وضع حالة المنتج أو وقت التشغيل تحت `.harness/`.** `./.harness/` مخصص للمصدر طويل العمر القابل للمراجعة. سجلات التفعيل وهاشات الانحراف وذاكرات المنتج المؤقتة تنتمي إلى مجلدات يملكها المنتج مثل `.harnex/` وإلى `.harnessIgnore`.
- **وضع ملفات جذر الهدف في حزمة بالخطأ.** ملف جذر هدف مثل `.claude/settings.json` ينبغي تمثيله عند `.harness/resources/.claude/settings.json`، لا داخل مجلد مهارة ولا داخل مجموعة موارد غير مرتبطة. صرّح به قابلاً للتعديل عندما يجب أن يُزرع مرة واحدة ثم يصبح مملوكاً لوقت التشغيل.
- **استخدام التجاوزات لتفريع المحتوى على نطاق واسع.** تستبدل مجلدات التجاوز مسارات نسبية دقيقة أو تضيف ملفات جديدة. هي أداة صغيرة عمداً؛ إذا احتاج هدف نسخة مختلفة جداً من مهارة، ففضل مورداً منفصلاً على شجرة تجاوز عميقة.
- **الالتزام بملفات كتبها وقت التشغيل كما لو كانت مصدراً.** ملفات مثل `.claude/settings.local.json` يجب غالباً نسخها إلى `.harness` كبذرة وإعلانها في `.harnessMutable` حتى يزرعها الإسقاط مرة واحدة ثم يتركها. إذا أصبح الملف لاحقاً سياسة مشتركة، فارفع البايتات المرغوبة إلى جذر المصدر المصرح به وأعد إسقاط الملف القابل للتعديل بقصد.
- **توقع ملف تجاهل في مخرجات الهدف قبل وجوده.** لا يشارك ملف `.harnessIgnore` في مخرجات الهدف إلا بعد وجوده على القرص. استخدم ملف الجذر للقواعد التي يجب أن تطبق في أول تفعيل.
- **إسقاط ضوابط Harness config كحمولة.** ملفات التصريح مثل `.harnessIgnore` و`.harnessMutable` و`.harnessProfile` و`.harnessProfileRoot` تُقرأ كضوابط ولا تُنسخ إلى الأهداف كملفات مُدارة.
- **استخدام الروابط الرمزية للأهداف.** لا يتبع Harness config v1 الروابط الرمزية. إذا شغل رابط رمزي مساراً يحتاج التفعيل إلى الكتابة فيه، يبلغ التفعيل عن تعارض افتراضياً. استبدل الرابط يدوياً، أو اضبط `[activation].targetSymlinks = "replace"`، أو استخدم `--replace-target-symlinks` فقط عندما يكون استبدال الرابط نفسه مقصوداً.

## تذكير بالنطاق

يبقى المعيار نفسه محدوداً بتخطيط المصدر، وتصريحات الأهداف، والتجاوزات، وتجاهلات الإسقاط، والملفات القابلة للتعديل، وإسقاط النسخ الحتمي. أما الاختيار الخاص بالمنتج، والأسواق، ومراجعة تعديلات الهدف، وسجلات التفعيل، والالتقاط، والمزامنة البعيدة، فمكانها فوق المعيار الأساسي.
